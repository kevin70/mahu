本文是当前阶段的电商商品与库存域设计稿，用于指导后续 Liquibase、Entity、DAL 与业务服务实现。

## 背景

当前仓库已经沉淀了独立 schema 的设计方式，例如 `iam` 与 `wallet`。商品、SKU、库存、预占、价格历史同样属于边界清晰且后续会持续扩展的业务域，不适合继续零散挂到其他 schema 中。

本方案引入独立 `ecommerce` schema，统一承载电商商品与库存相关数据，并明确以下当前约束：

- 当前按单商家系统设计，不引入 `merchant_id`
- 当前商品主档使用 `product` 命名，语义上等同传统电商中的 SPU
- 当前范围覆盖商品目录、复杂属性、SKU、多仓库存、库存预占、价格历史
- 当前不覆盖采购、批次、效期、序列号、订单正式表

## 设计目标

本方案需要同时满足以下目标：

- 支持 `product -> product_sku` 两层商品模型
- 支持类目、品牌、复杂属性模板与销售属性组合
- 支持多仓库存
- 支持下单预占、支付确认扣减、取消或超时释放
- 支持 SKU 当前价格与价格变更历史
- 保持 schema、表名、主键、约束与事务边界清晰稳定
- 为后续扩展订单、供应链、促销价保留空间，但不在首版过度设计

## 总体边界

### `ecommerce` 的职责边界

- `ecommerce`：商品目录、商品属性、SKU、媒体、价格历史、仓库、库存、库存预占、库存流水
- 外部订单域：通过 `biz_type + biz_id` 关联库存预占与扣减，不在本 schema 内定义订单主表
- 供应链域：采购、入库单、批次、效期、调拨等能力后续独立扩展，不进入本期核心表

### 统一建模原则

- `product` 表示商品主档，不直接承担库存扣减
- `product_sku` 表示最小可售卖单元，价格和库存都挂在 SKU 层
- 结构化可筛选属性进入独立属性表，低频扩展字段进入 `JSONB`
- 库存当前态与库存流水分离，所有库存动作都必须落流水
- 库存控制采用“预占 -> 确认扣减 -> 释放”的事务模型
- 所有电商表放入 `ecommerce` schema，SQL 与映射显式写全 schema 前缀，不依赖 `search_path`

## 命名与通用约定

### Schema 约定

- 独立 schema 名固定为 `ecommerce`
- 所有 DDL、DML、索引、FK、视图、手写 SQL 显式使用 `ecommerce.xxx`
- 不依赖数据库 `search_path`

### 主键与时间字段

- 主表主键统一使用 `BIGINT GENERATED ALWAYS AS IDENTITY`
- 创建时间统一使用 `created_at TIMESTAMPTZ`
- 当前态表统一保留 `updated_at TIMESTAMPTZ`
- 大多数主表推荐保留 `created_at` / `updated_at`

### 数值与状态字段

- 金额字段统一使用 `NUMERIC(18,2)`
- 数量字段统一使用 `BIGINT`
- 低频扩展字段使用 `JSONB`
- 本文中的 `status`、`publish_status`、`sale_status` 先以语义状态表达
- 若在 Mahu 中正式落地，建议优先映射为项目统一的整数状态码，并对齐 [元数据规范](/architecture/metadata-dictionary)

### 当前不引入的字段

以下字段当前不进入核心表：

- `merchant_id`
- 批次号、效期、序列号
- 促销价、会员价、阶梯价
- 订单头与订单行正式表

首版 `txn_type` 建议至少支持：

- `inbound`
- `reserve`
- `release`
- `deduct`
- `adjust`
- `transfer_out`
- `transfer_in`

约束与索引建议：

- 主键：`inventory_txn_pk`
- 外键：`warehouse_id -> ecommerce.warehouse.id`
- 外键：`sku_id -> ecommerce.product_sku.id`
- 普通索引：`warehouse_id, sku_id, created_at desc`
- 普通索引：`biz_type, biz_id`
- 普通索引：`created_at desc`

说明：

- 库存流水采用追加写，不做更新覆盖
- 任意库存变动都必须落一条流水，便于审计与追溯

## 库存事务规则

### 创建预占

执行原则：

1. 锁定目标 `ecommerce.warehouse_inventory` 行
2. 校验 `available_qty >= requested_qty`
3. 更新 `reserved_qty += requested_qty`
4. 写入 `ecommerce.inventory_reservation`
5. 写入 `ecommerce.inventory_reservation_item`
6. 写入 `ecommerce.inventory_txn`，`txn_type = reserve`

### 确认预占

执行原则：

1. 锁定对应库存行
2. 更新 `on_hand_qty -= confirmed_qty`
3. 更新 `reserved_qty -= confirmed_qty`
4. 更新预占单状态为 `confirmed`
5. 写入 `ecommerce.inventory_txn`，`txn_type = deduct`

### 释放预占

执行原则：

1. 锁定对应库存行
2. 更新 `reserved_qty -= released_qty`
3. 更新预占单状态为 `released` 或 `expired`
4. 写入 `ecommerce.inventory_txn`，`txn_type = release`

### 幂等与一致性要求

- 已过期预占单不可再次确认
- 同一业务单不可重复创建第二张有效预占单
- 重复确认、重复释放不能导致重复扣减或重复回补
- 应用服务层负责事务编排，数据库层负责唯一约束、数量约束和流水留痕

## 测试与校验建议

数据库与服务实现阶段至少覆盖以下场景：

- `product_code`、`sku_code` 唯一约束生效
- 同一 `product` 下重复销售属性组合无法创建第二个 SKU
- 同一 `(warehouse_id, sku_id)` 只能有一条库存当前态
- `reserved_qty` 永远不大于 `on_hand_qty`
- 预占后 `available_qty` 正确下降
- 确认预占后 `on_hand_qty` 与 `reserved_qty` 同步变化正确
- 释放预占后 `reserved_qty` 正确回补
- 并发预占同一库存时不出现超卖
- 改价会新增一条 `sku_price_history`
- 每次库存动作都会新增一条 `inventory_txn`
- 不设置 `search_path` 时，所有 SQL 与 ORM 映射仍能正确访问 `ecommerce` schema

## 当前假设与暂不覆盖范围

当前假设：

- 当前系统为单商家，不引入 `merchant_id`
- `product` 在语义上等同传统 SPU
- 外部订单域未来通过 `biz_type + biz_id` 与库存预占关联

本期暂不覆盖：

- 订单主表与订单行正式表
- 采购、入库、调拨、批次、效期、序列号
- 促销价、会员价、阶梯价、未来生效价
- 按时间分区的历史表方案

如果后续 `inventory_txn` 或 `sku_price_history` 数据量明显增长，再评估按时间做 RANGE 分区。
