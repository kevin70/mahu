## 环境名称

| 简写   | 全称                         | 用途                       |
|------|----------------------------|--------------------------|
| DEV  | Development Environment    | 开发环境，开发者本地或共享编码调试环境      |
| SIT  | System Integration Testing | 系统集成测试环境，验证多个模块/服务间的交互   |

## 数据状态

本节定义 Mahu 的**全局唯一状态码注册表**。这份注册表回答三个问题：

- 哪些信息适合放进 `status`
- 状态码应该如何命名和分段
- 各领域应该如何从主表中选状态子集

### 基本原则

1. 状态码在项目范围内**全局唯一**，一个编码只允许对应一个稳定语义。
2. 具体领域状态不是重新发明一套编码，而是从注册表中选择一个**子集**。
3. 不同领域只有在状态语义完全相同的情况下才能复用同一个状态码。
4. 若现有主表无法表达业务语义，必须先更新本文档，再进入数据库迁移、OpenAPI、代码枚举与字典配置。

### 使用边界

适合进入 `status` 的信息：

- 生命周期阶段，例如 `PENDING`、`PROCESSING`、`COMPLETED`
- 领域无关且语义稳定的结果态，例如 `FAILED`、`CANCELLED`、`EXPIRED`
- 可以直接驱动查询、索引、权限或业务分支的稳定状态

不建议直接进入 `status` 的信息：

- 删除标记、可见性、启用开关等已有独立字段表达的信息
- 失败原因、取消原因、异常来源等“原因”类信息
- 可以由其他字段稳定推导的状态，例如通知是否过期可优先由 `expireAt` 推导

### 选用流程

1. 先判断字段表达的是不是“生命周期状态”，而不是原因、模式或标记。
2. 如果是状态，从本文注册表中选择最贴近的已有语义。
3. 如果没有合适语义，先更新本文档，再进入数据库、OpenAPI 和代码实现。

### 编码段位

| 区间   | 阶段       | 说明                         |
|------|----------|----------------------------|
| 100-199 | 初始/等待态   | 新建、待处理、待审核、待支付等尚未真正开始流转的状态 |
| 200-299 | 通用进行态    | 已生效、进行中、处理中、可用、已确认等领域无关的进行中状态 |
| 300-399 | 交易/支付态   | 支付、退款等交易生命周期专用状态            |
| 400-499 | 履约/物流态   | 打包、发货、运输、送达等履约链路专用状态       |
| 700-799 | 限制态      | 禁用、锁定、冻结等受限但未终结的状态         |
| 800-899 | 正常终态     | 完成、归档、结算等“成功结束”或“正常收束”的状态 |
| 900-999 | 异常/失效终态 | 失败、过期、取消、终止等不可继续流转的状态      |

### 状态命名规范

1. 状态名统一使用 `UPPER_SNAKE_CASE`。
2. 同一语义只保留一个英文名称，禁止并存同义词。
3. 等待某个前置条件的子状态统一使用 `PENDING_*` 前缀。
4. 通用进行态优先使用最简单且稳定的名称：
   `ACTIVE`、`PROCESSING`、`RUNNING`。
5. 已完成或已结束的状态优先使用形容词或过去分词：
   `APPROVED`、`CONFIRMED`、`COMPLETED`、`CANCELLED`、`EXPIRED`。
6. 只有当状态语义稳定且可跨项目复用时，才允许进入本注册表。

### 领域选值示例

以下不是重新定义编码，而是说明常见领域应如何**从主表中选子集**。本节只保留常用领域对象示例，且同一状态语义只给出一组推荐值。

#### AdminStatus

推荐取值：

- `ACTIVE(200)`：管理员可登录、可正常使用
- `DISABLED(720)`：管理员被主动禁用

#### DelayedTaskStatus

推荐取值：

- `PENDING(130)`：已入队，等待被 worker 领取
- `PROCESSING(220)`：已被 worker 领取并处理
- `COMPLETED(800)`：执行成功
- `ARCHIVED(810)`：完成后归档
- `FAILED(900)`：重试耗尽或被判定最终失败

#### NotificationStatus

推荐取值：

- `ACTIVE(200)`：通知当前有效
- `EXPIRED(920)`：若必须落库时使用

说明：若 `expireAt` 足以表达通知是否过期，优先只落库 `ACTIVE(200)`，通过时间推导过期语义。

#### PaymentStatus

推荐取值：

- `PENDING_PAYMENT(190)`
- `PAID(300)`
- `REFUNDING(320)`
- `REFUNDED(330)`

#### ApprovalStatus

推荐取值：

- `PENDING_REVIEW(180)`
- `APPROVED(210)`
- `REJECTED(910)`

### 扩展规则

1. 单个字段通常只应选择 2-6 个状态，不要为了“完整”把无关状态全部暴露给前端和数据库。
2. 默认优先从“最常用状态”中选；只有在语义不够用时再选择扩展状态。
3. 优先选择语义最贴近的主状态，不要把 `PENDING`、`PROCESSING`、`RUNNING`、`ACTIVE` 混用。
4. `PROCESSING` 偏系统处理，`RUNNING` 偏长时间运行中的任务或进程。
5. 同一字段应尽量只保留一个“等待态”语义入口，避免同时混用 `PENDING` 与其他近义等待态。
6. 数据库 `CHECK` 约束、字典分组、OpenAPI 枚举说明必须与本节保持一致。
7. 任何新增状态都必须先更新本文档，再进入数据库迁移、代码枚举与接口契约。

### 落地要求

本文档即当前生效的状态规范。

1. 数据库、代码、OpenAPI、字典数据统一采用本文三位数状态码注册表。
2. 新字段、新模型、新接口必须直接从本文注册表中选值。
3. 任何新增状态都必须先更新本文档，再落地到数据库、接口契约和代码实现。
4. 同一领域的查询条件、索引条件、默认值、字典项与接口说明必须保持一致。

## 数据可见性

| 值 | 英文名称         | 中文名称 | 说明                                                                  |
|---|--------------|------|---------------------------------------------------------------------|
| 0 | `PRIVATE`    | 私有   | 只有内容所有者或特定用户（如被授权者）可以查看。通常用于个人数据、私密内容                               |
| 1 | `PUBLIC`     | 公开   | 任何人都可以查看，无需登录或特定权限。通常用于希望广泛传播的内容                                    |
| 2 | `UNLISTED`   | 未列出  | 内容可以被任何人查看，但不会出现在公共列表、搜索结果或推荐中。只有拥有链接的人可以访问。通常用于分享给特定人群但不希望完全公开的内容。 |
| 3 | `RESTRICTED` | 受限   | 内容对特定群体或满足特定条件的用户可见。需要一定的权限或满足条件（如年龄、地理位置、用户组等）                     |
