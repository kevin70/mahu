## 总体规则

数据库对象命名统一使用 `snake_case`，并遵循“短、稳、可预测”的原则：

1. 表名、列名、索引名、约束名全部使用小写下划线风格。
2. 名称优先表达业务含义，不为了“通用”而使用模糊词。
3. 同类对象保持统一模式，避免同一仓库内混用多套命名习惯。
4. 新增命名时，优先对齐已有表与索引模式，而不是重新发明一套规则。

## Schema 与表名

### Schema

- 业务表默认按领域归入 `sys` 等业务 schema。
- 全局字典与跨领域基础元数据可放在 `public`。

当前仓库示例：

- `sys.admins`
- `sys.admin_notifications`
- `sys.delayed_tasks`
- `public.dict_groups`
- `public.dicts`

### 表名

- 普通业务表使用**复数名词**：`admins`、`roles`、`feature_flags`
- 关联表使用 `left_rights` 形式：`admin_roles`
- 明细/流水/日志表使用主体名 + 类型后缀：`admin_auth_logs`、`scheduled_task_logs`
- 定向/已读这类从属表使用主体名 + 子语义：`admin_notification_targets`、`admin_notification_reads`

不建议：

- 使用单数表名，如 `admin`、`role`
- 使用无业务含义的缩写表名
- 同一领域里同时出现复数和单数混用

## 列命名

### 通用字段

以下字段名优先复用，不要在相同语义上起新名字：

- `id`：主键
- `created_at`：创建时间
- `updated_at`：更新时间
- `deleted`：软删除标记
- `status`：状态码，取值规则见「[元数据规范](/architecture/metadata-dictionary)」
- `type`：类型码
- `code`：业务代码
- `name`：名称
- `title`：标题
- `description`：描述
- `label`：展示文案
- `value`：值
- `ordering`：排序值
- `payload`：JSON 载荷
- `metadata`：扩展元数据

### 外键与关联字段

- 外键统一使用 `{entity}_id`
- 如果同表内存在多个同类外键，必须保留业务语义前缀

示例：

- `admin_id`
- `notification_id`
- `group_id`
- `change_log_id`

不建议：

- 只写 `id2`、`ref_id2` 这类无语义名称
- 在明显不同语义下都统称为 `user_id`

### 时间字段

- 时间点统一使用 `_at` 后缀：`created_at`、`updated_at`、`expire_at`
- 时间范围统一使用 `_from` / `_to` 或领域明确的时间点命名
- 调度/执行场景优先用可表达业务语义的具体字段名，如 `execution_time`、`last_heartbeat`、`delay_until`

### 布尔字段

数据库布尔字段默认不强制加 `is_` / `has_` 前缀，优先遵循现有表风格。

当前仓库常见风格：

- `enabled`
- `deleted`
- `preset`

只有当字段语义不清时，才考虑 `is_` / `has_` 前缀。

### 状态与类型字段

- 状态字段统一命名为 `status`
- 类型字段统一命名为 `type`
- 若某字段不是通用状态码，而是某个领域的局部状态，也仍建议保持 `{domain}_status` 用于**参数/模型命名**，表字段本身优先仍叫 `status`

示例：

- 表字段：`status`
- 查询参数：`order_status`
- DTO 字段：`payment_type`

## 约束与索引命名

### 主键/唯一约束

- 主键约束：`{table}_pk`
- 唯一索引：`{table}_{columns}_uidx`

示例：

- `admins_pk`
- `dicts_pk`
- `admins_username_uidx`
- `dicts_group_id_value_uidx`

### 普通索引

- 普通索引：`{table}_{columns}_idx`
- 多列索引按列顺序展开
- 带过滤条件的部分索引也沿用相同模式，不把条件硬编码进索引名

示例：

- `admin_auth_logs_admin_id_idx`
- `scheduled_tasks_execution_time_idx`
- `admin_notifications_type_created_at_idx`

不建议：

- 使用 `idx_foo` 这类前缀式命名
- 使用 `index1`、`tmp_idx` 这类临时名称
- 在同一仓库里混用 `idx_table_col` 和 `table_col_idx`

## 统计表规范

统计表、汇总表、快照表应与业务明细表区分开来，命名和建模都要体现“聚合结果”属性。

### 表名规则

- 通用统计表优先使用 `{subject}_stats`
- 按时间粒度拆分的统计表优先使用 `{subject}_{grain}_stats`
- 快照表优先使用 `{subject}_snapshots` 或 `{subject}_{grain}_snapshots`

推荐示例：

- `admin_stats`
- `feature_flag_daily_stats`
- `order_hourly_stats`
- `user_snapshots`

不建议：

- 使用 `report`、`summary`、`data` 这类过于宽泛的后缀
- 在统计表名中混入实现细节，如 `tmp_stats`、`cache_stats`
- 明细表和统计表名称过近，导致一眼看不出一个是事实表，一个是聚合表

### 统计表字段规则

- 维度字段继续沿用普通表命名，例如 `admin_id`、`group_id`、`type`
- 时间粒度字段应显式命名，不要只保留 `created_at`
- 聚合指标字段统一使用可读的后缀

推荐时间字段：

- `stat_date`：按天统计
- `stat_hour`：按小时统计
- `stat_month`：按月统计
- `snapshot_at`：快照时间点

推荐指标字段：

- `*_count`：计数，如 `login_count`、`error_count`
- `*_amount`：金额或累计值，如 `paid_amount`
- `*_avg`：平均值，如 `response_time_avg`
- `*_max` / `*_min`：极值
- `*_rate`：比率，如 `success_rate`

### 统计表约束与索引

- 主键仍优先使用 `id`，但如果业务上明确需要维度唯一，也可以用唯一索引约束粒度
- 统计表至少应有一组“维度 + 时间粒度”唯一约束或唯一索引
- 时间过滤索引优先覆盖 `stat_date` / `stat_hour` 等粒度字段，而不是只依赖 `created_at`

推荐示例：

- `feature_flag_daily_stats_feature_flag_id_stat_date_uidx`
- `order_hourly_stats_stat_hour_idx`

## 常用字段命名参考

优先使用仓库里已经稳定出现或足够明确的字段词汇：

- 标识类：`id`、`code`
- 展示类：`name`、`title`、`label`、`remark`
- 时间类：`created_at`、`updated_at`、`expire_at`
- 状态类：`status`、`type`
- 关系类：`admin_id`、`notification_id`、`group_id`
- 扩展类：`payload`、`metadata`

可继续复用的常见字段参考：

- 审计类：`created_by`、`updated_by`
- 排序类：`ordering`
- 金额类：`amount`、`total_amount`、`refund_amount`
- 计数类：`view_count`、`click_count`、`download_count`
- 比率类：`success_rate`、`conversion_rate`
- 文本类：`description`、`content`
- 标记类：`enabled`、`deleted`、`preset`

不再建议继续扩展一大批“通用词典式”列名清单。新增字段时应优先从真实业务语义出发，并对齐现有表。

## 建模建议

设计新表或新增列时，按下面顺序检查：

1. 先看仓库里同类表是否已有现成命名模式。
2. 状态字段是否应复用 `status`，并遵守状态码注册表。
3. 外键是否能直接落成 `{entity}_id`。
4. 时间字段是否应统一成 `_at`。
5. 索引和约束名称是否符合 `{table}_{columns}_{suffix}` 模式。

如果一个名称需要额外解释才能看懂，通常说明它还不够好。

## 代表性示例

下面几组示例覆盖 Mahu 中最常见的几类表设计，可作为新表命名时的直接参考。

### 示例一：业务主表

适用于“一个聚合对应一张主表”的场景，例如管理员、角色、功能开关。

```sql
CREATE TABLE sys.admins (
  id serial CONSTRAINT admins_pk PRIMARY KEY,
  created_at timestamptz,
  updated_at timestamptz,
  deleted boolean default false,
  username varchar(128),
  nickname varchar(128),
  avatar varchar(128),
  status smallint
);

CREATE UNIQUE INDEX admins_username_uidx ON sys.admins (username)
WHERE
  (deleted = false);
```

命名要点：

- 表名使用复数：`admins`
- 主键约束使用 `{table}_pk`
- 通用字段统一使用 `created_at`、`updated_at`、`deleted`、`status`
- 唯一索引使用 `{table}_{columns}_uidx`

### 示例二：流程/任务表

适用于有明确生命周期和调度语义的流程型表，例如延时任务、调度任务。

```sql
CREATE TABLE sys.delayed_tasks (
  id uuid NOT NULL CONSTRAINT delayed_tasks_pk PRIMARY KEY,
  created_at timestamptz,
  updated_at timestamptz,
  topic varchar(50),
  status smallint,
  delay_until timestamptz,
  attempts integer,
  max_attempts smallint,
  lock_at timestamptz,
  lease_seconds smallint,
  reference_id varchar(128) NOT NULL,
  payload jsonb,
  idempotency_key varchar(128)
);

CREATE UNIQUE INDEX delayed_tasks_topic_idempotency_key_uidx
  ON sys.delayed_tasks (topic, idempotency_key);

CREATE INDEX delayed_tasks_delay_until_idx
  ON sys.delayed_tasks (delay_until)
WHERE
  (status = 130);
```

命名要点：

- 表名直接表达业务对象：`delayed_tasks`
- 流程字段使用具体语义名：`delay_until`、`attempts`、`lock_at`
- 外部关联字段不用模糊命名，直接写 `reference_id`
- 部分索引名称仍按 `{table}_{columns}_idx`，不把过滤条件写进名称

### 示例三：从属/关联表

适用于“主表 + 定向目标/已读记录/关系表”这类从属结构。

```sql
CREATE TABLE sys.admin_notification_targets (
  id bigint generated always as identity CONSTRAINT admin_notification_targets_pk PRIMARY KEY,
  notification_id bigint NOT NULL,
  admin_id integer NOT NULL,
  created_at timestamptz
);

CREATE UNIQUE INDEX admin_notification_targets_notification_id_admin_id_uidx
  ON sys.admin_notification_targets (notification_id, admin_id);

CREATE INDEX admin_notification_targets_admin_id_idx
  ON sys.admin_notification_targets (admin_id);
```

命名要点：

- 表名保留主体 + 子语义：`admin_notification_targets`
- 外键统一使用 `{entity}_id`
- 多列唯一索引按字段顺序展开

### 示例四：统计表

适用于按时间粒度聚合后的统计表。

```sql
CREATE TABLE sys.feature_flag_daily_stats (
  id bigint generated always as identity CONSTRAINT feature_flag_daily_stats_pk PRIMARY KEY,
  feature_flag_id integer NOT NULL,
  stat_date date NOT NULL,
  hit_count integer NOT NULL DEFAULT 0,
  miss_count integer NOT NULL DEFAULT 0,
  error_count integer NOT NULL DEFAULT 0,
  success_rate numeric(5, 2),
  created_at timestamptz,
  updated_at timestamptz
);

CREATE UNIQUE INDEX feature_flag_daily_stats_feature_flag_id_stat_date_uidx
  ON sys.feature_flag_daily_stats (feature_flag_id, stat_date);

CREATE INDEX feature_flag_daily_stats_stat_date_idx
  ON sys.feature_flag_daily_stats (stat_date);
```

命名要点：

- 表名使用 `{subject}_{grain}_stats`
- 时间粒度字段使用 `stat_date`
- 指标字段统一使用 `*_count`、`*_rate`
- 唯一索引明确表达“维度 + 粒度”
