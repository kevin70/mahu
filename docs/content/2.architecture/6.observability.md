## traceId 与 x-request-id

应用在 HTTP 层通过以下方式处理追踪 ID：

- 入站请求优先读取 header `x-request-id`，若不存在则生成新的 traceId。
- 将 traceId 写入 Helidon MDC，键为 `G.MDC_TRACE_ID`。
- 在响应阶段，通过 `x-request-id` header 将 traceId 返回给调用方。

约定如下：

1. **上游透传优先**：如果上游已经设置了 `x-request-id`，应用应保持原值，不要重新生成。
2. **内部生成兜底**：仅在 header 缺失时才生成新的 traceId。
3. **出站调用透传**：在进行 HTTP 出站调用时，应将当前 traceId 作为 `x-request-id`（或团队统一约定的 correlation header）传递给下游服务。

## 错误响应规范

应用通过统一的错误处理器（如 `RestErrorHandler`）将异常转换为标准化的 HTTP 响应。建议在以下几个层面达成一致：

- **结构**：错误响应返回结构固定为 `{"error": ...}`，其中 `error` 对象包含至少以下字段（字段名与类型需与实现一致）：
  - `trace_id`：追踪 ID（`ProblemResponse` 使用 `@JsonProperty("trace_id")`）。
  - `timestamp`：错误发生时间（ISO 8601 / 带偏移）。
  - `path`：请求路径。
  - `method`：HTTP 方法。
  - `status`：HTTP 状态码。
  - `code`：业务错误码（整数）。
  - `message`：错误描述。
  - 可选：`stacktrace`（仅非 PROD 环境返回）。
- **details 的落地方式**：实现中 `ProblemResponse.details` 使用 `@JsonUnwrapped`，因此 `details` 内的 key 会被直接展开到 `error` 对象同级（例如 `exception`、`exception_message`、`invalid_params` 等），而不是嵌套在 `details` 字段下。
- **OpenAPI 描述**：
  - 在 `openapi/admin/components/schemas/ErrorResponse.yaml` 中定义统一的 `ErrorResponse` schema。
  - 在各个接口的 `4xx`/`5xx` 或 `default` 响应中引用该 schema，确保文档与实际返回结构保持一致。

当调整错误响应结构或字段含义时，应同步更新 OpenAPI 文档和实现代码，避免前后端或服务间出现理解偏差。

最小示例（仅展示关键字段）：

```json
{
  "error": {
    "trace_id": "2zy6qqr5mq8zljpxv7g8s",
    "timestamp": "2025-09-19T21:17:00+08:00",
    "path": "/samples",
    "method": "POST",
    "status": 400,
    "code": 140003,
    "message": "缺少参数",
    "exception": "java.lang.IllegalArgumentException",
    "exception_message": "bad"
  }
}
```
