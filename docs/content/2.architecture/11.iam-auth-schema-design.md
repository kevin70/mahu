本文是当前阶段的认证域设计稿，用于指导后续 Liquibase、Entity、DAL 与业务服务实现。

## 背景

当前仓库已经存在系统侧认证配置与管理员认证能力：

- `sys.auth_clients`：承载登录渠道配置，如微信 `appid/appsecret`
- `sys.admins`、`sys.admin_login_logs`：承载管理端认证主体与登录日志

业务用户认证不应继续混入 `sys`。原因如下：

1. `sys` 更适合放系统管理数据，而不是面向业务用户的身份数据。
2. 用户认证会继续扩展第三方提供方、会话管理、审计、风控，不适合挂在管理员模型旁边。
3. 若把微信、华为等厂商字段直接塞进用户主表，后续每增加一个提供方都要继续改表。

因此本方案引入独立 `iam` schema，统一承载业务用户认证域。

## 设计目标

本方案需要同时满足以下目标：

- 支持手机号密码登录
- 支持微信应用授权登录
- 支持华为应用授权登录
- 允许一个用户绑定多个第三方身份
- 支持登录日志审计
- 支持刷新令牌会话管理
- 避免为单一厂商设计专属核心字段
- 为后续接入 Apple、Google、支付宝等提供方预留扩展位

## 总体边界

### `iam` 与 `sys` 的职责边界

- `iam`：业务用户主体、登录凭证、第三方身份绑定、登录日志、会话
- `sys`：系统管理配置、管理端用户、管理端权限、登录渠道配置

当前阶段保留对 `sys.auth_clients.client_id` 的跨 schema 引用，原因是登录渠道配置已经在 `sys` 中落地，短期内没有必要重复建表。

### 统一建模原则

- 用户主表只保存稳定用户资料和账号状态
- 本地密码等认证因子独立入表
- 第三方授权身份统一入绑定表，不为微信、华为单独建核心表
- 第三方特有附加信息进入 `JSONB` 扩展字段，不进入主查询键
- access token 不入库，会话只持久化 refresh token 对应的 session
- 敏感凭据只保存摘要、引用或加密值，不保存明文
- 首版不建立数据库外键，关联完整性由应用层与业务校验保证
- 低基数、稳定枚举语义的字段优先使用 `SMALLINT`
- 业务编码、主体标识、展示文案、UA 等结构化短文本优先使用明确长度的 `VARCHAR(n)`，不默认使用 `TEXT`
- 关联字段长度应与目标标识保持一致，例如 `client_id` 统一跟随 `sys.auth_clients.client_id`
- 索引优先服务唯一约束与高频命中查询，不为潜在分析或低频管理查询预建大而全索引

## Schema 总览

| 表名 | 用途 |
|---|---|
| `iam.users` | 用户主体表 |
| `iam.user_credentials` | 本地登录凭证表 |
| `iam.user_identities` | 第三方身份绑定表 |
| `iam.login_logs` | 登录日志表 |
| `iam.user_sessions` | 刷新令牌会话表 |

## 表设计

### `iam.users`

用户主体表，只承载与登录方式无关的稳定数据。

推荐字段：

| 字段 | 类型 | 说明 |
|---|---|---|
| `id` | `BIGINT` | 主键 |
| `created_at` | `TIMESTAMPTZ` | 创建时间 |
| `updated_at` | `TIMESTAMPTZ` | 更新时间 |
| `deleted` | `BOOLEAN` | 软删除标记 |
| `status` | `SMALLINT` | 用户状态，建议使用全局状态码 |
| `nickname` | `VARCHAR(128)` | 昵称 |
| `avatar` | `VARCHAR(512)` | 头像地址 |
| `mobile` | `VARCHAR(32)` | 手机号 |
| `mobile_verified_at` | `TIMESTAMPTZ` | 手机号验证时间 |
| `last_login_at` | `TIMESTAMPTZ` | 最近登录时间 |

约束与索引建议：

- 主键：`users_pk`
- `mobile` 对 `deleted = false` 建部分唯一索引 `users_mobile_uidx`
- `status` 非空，默认建议使用 `ACTIVE(200)`
- `deleted` 非空，默认 `false`

不进入本表的字段：

- 密码
- 微信 `openid`
- 微信 `unionid`
- 华为用户唯一标识
- access token / refresh token

### `iam.user_credentials`

本地认证凭证表，用于手机号密码等本地认证能力。

推荐字段：

| 字段 | 类型 | 说明 |
|---|---|---|
| `id` | `BIGINT` | 主键 |
| `user_id` | `BIGINT` | 关联用户 |
| `credential_type` | `VARCHAR(32)` | 凭证类型 |
| `principal` | `VARCHAR(128)` | 登录主体，如手机号 |
| `secret` | `BYTEA` | 密码摘要或加密值 |
| `secret_alg` | `VARCHAR(32)` | 密码算法标识 |
| `enabled` | `BOOLEAN` | 凭证是否启用 |
| `verified_at` | `TIMESTAMPTZ` | 凭证验证通过时间 |
| `expire_at` | `TIMESTAMPTZ` | 凭证失效时间 |
| `created_at` | `TIMESTAMPTZ` | 创建时间 |
| `updated_at` | `TIMESTAMPTZ` | 更新时间 |

首版推荐仅支持以下类型：

- `mobile_password`

约束与索引建议：

- 唯一索引：`(credential_type, principal)`
- `enabled` 非空，默认 `true`

说明：

- `secret` 只保存密码摘要或加密结果，不保存明文
- `secret` 延续仓库现有敏感凭据列的 `BYTEA` 风格，便于与加密或摘要结果对齐
- 若后续接入邮箱密码、用户名密码，只需扩展 `credential_type`

### `iam.user_identities`

第三方身份绑定表，统一承载微信、华为及后续其他提供方的授权身份。

推荐字段：

| 字段 | 类型 | 说明 |
|---|---|---|
| `id` | `BIGINT` | 主键 |
| `user_id` | `BIGINT` | 关联用户 |
| `provider` | `VARCHAR(32)` | 提供方类型 |
| `provider_app_id` | `VARCHAR(128)` | 第三方应用标识 |
| `provider_subject` | `VARCHAR(191)` | 第三方主体唯一标识 |
| `provider_union_id` | `VARCHAR(191)` | 跨应用统一主体标识，可为空 |
| `client_id` | `VARCHAR(50)` | 本系统登录渠道 ID |
| `scope` | `VARCHAR(512)` | 授权范围 |
| `profile` | `JSONB` | 第三方附加资料 |
| `last_login_at` | `TIMESTAMPTZ` | 最近一次授权登录时间 |
| `created_at` | `TIMESTAMPTZ` | 绑定关系创建时间 |
| `updated_at` | `TIMESTAMPTZ` | 更新时间 |
| `deleted` | `BOOLEAN` | 软删除标记 |

首版 `provider` 推荐值：

- `wechat_xcx`
- `wechat_web`
- `huawei_app`

字段映射规则：

- `provider_app_id`
  - 微信：`appid`
  - 华为：应用 `client_id` 或 `app_id`
- `provider_subject`
  - 微信：`openid`
  - 华为：厂商返回的用户唯一主体标识，如 `openid` 或 `sub`
- `provider_union_id`
  - 微信：`unionid`
  - 华为：若无等价概念则为空

约束与索引建议：

- 唯一索引：`(provider, client_id, provider_subject)`
- `deleted` 非空，默认 `false`

设计原则：

- 不新增 `wechat_openid`、`wechat_unionid`、`huawei_openid` 这类专属核心列
- 统一通过 `provider + provider_subject` 表达第三方登录主体
- `profile` 仅保存附加资料，不作为核心查询条件
- `client_id` 长度统一跟随 `sys.auth_clients.client_id = VARCHAR(50)`
- `created_at` 即该绑定关系的首次绑定时间，不再单独保留 `linked_at`
- 首版按用户查看绑定列表不是高频主路径，因此不单独预建 `user_id`、`client_id`、`provider` 单列索引

### `iam.login_logs`

登录日志表，用于记录每次登录结果，并兼顾基础审计与风控检索。

推荐字段：

| 字段 | 类型 | 说明 |
|---|---|---|
| `id` | `UUID` | 主键 |
| `created_at` | `TIMESTAMPTZ` | 创建时间 |
| `user_id` | `BIGINT` | 关联用户，可为空 |
| `identity_id` | `BIGINT` | 关联第三方身份，可为空 |
| `grant_type` | `VARCHAR(50)` | 授权类型 |
| `provider` | `VARCHAR(32)` | 提供方类型 |
| `client_id` | `VARCHAR(50)` | 登录渠道 ID |
| `principal_masked` | `VARCHAR(191)` | 脱敏后的登录主体 |
| `principal_hash` | `BYTEA` | 登录主体归一化摘要 |
| `success` | `BOOLEAN` | 是否成功 |
| `reason_code` | `VARCHAR(50)` | 失败原因码 |
| `reason_detail` | `VARCHAR(255)` | 可控失败摘要 |
| `ip_addr` | `VARCHAR(64)` | 来源 IP |
| `user_agent` | `VARCHAR(512)` | 请求 UA |
| `request_id` | `VARCHAR(26)` | 请求追踪 ID |

约束与索引建议：

- 主键：`login_logs_pk`
- `created_at` 非空，默认 `now()`
- `grant_type` 非空
- `success` 非空
- 首版普通索引：`(user_id, created_at DESC)`
- 首版部分索引：`(principal_hash, created_at DESC)`，条件 `principal_hash IS NOT NULL`

说明：

- `user_id` 与 `identity_id` 都可为空，覆盖认证失败且尚未定位到用户或绑定记录的场景
- `principal_masked` 用于展示与人工排查，例如脱敏手机号
- `principal_hash` 用于按登录主体做精确归并、风控判断和失败聚合，不写入明文主体
- `id`、`grant_type`、`reason_code`、`ip_addr` 的长度风格优先对齐现有 `sys.admin_login_logs` 与 `sys.admin_access_logs`
- `login_logs` 属于高写入追加表，首版索引应保持最小集合，避免为低频查询预建过多索引
- `identity_id`、`success`、`reason_code`、`client_id`、`provider`、`ip_addr` 不建议首版默认建索引，等明确出现慢查询或稳定分析维度后再补
- 日志表不做软删除，采用追加写入

### `iam.user_sessions`

刷新令牌会话表，用于续期、失效控制、设备管理和踢下线。

推荐字段：

| 字段 | 类型 | 说明 |
|---|---|---|
| `id` | `UUID` | 主键 |
| `user_id` | `BIGINT` | 关联用户 |
| `session_token` | `BYTEA` | refresh token 摘要 |
| `client_id` | `VARCHAR(50)` | 登录渠道 ID |
| `grant_type` | `VARCHAR(50)` | 授权类型 |
| `provider` | `VARCHAR(32)` | 会话来源 |
| `identity_id` | `BIGINT` | 关联第三方身份，可为空 |
| `ip_addr` | `VARCHAR(64)` | 来源 IP |
| `user_agent` | `VARCHAR(512)` | 请求 UA |
| `issued_at` | `TIMESTAMPTZ` | 签发时间 |
| `last_accessed_at` | `TIMESTAMPTZ` | 最近访问时间 |
| `expire_at` | `TIMESTAMPTZ` | 过期时间 |
| `revoked_at` | `TIMESTAMPTZ` | 失效时间 |
| `revoke_reason` | `VARCHAR(50)` | 失效原因 |

约束与索引建议：

- 主键：`user_sessions_pk`
- 唯一索引：`user_sessions_session_token_uidx`

说明：

- `session_token` 只保存 refresh token 摘要，不保存明文
- access token 不持久化入库
- `client_id` 长度统一跟随 `sys.auth_clients.client_id = VARCHAR(50)`
- 本地密码登录时 `identity_id` 可为空
- 第三方授权登录时可关联命中的 `user_identities.id`
- 按 `session_token` 摘要命中是首版核心路径；按用户列会话、过期扫描和后台管理筛选不是首版主场景，因此暂不预建 `user_id`、`expire_at`、`revoked_at` 相关索引

## 提供方适配规则

### 微信应用授权

推荐映射：

- `provider = wechat_xcx` 或 `wechat_web`
- `provider_app_id = appid`
- `provider_subject = openid`
- `provider_union_id = unionid`

注意事项：

- `unionid` 不是所有场景都稳定返回，首版不对其建立唯一约束
- 微信不同应用下的 `openid` 不同，因此唯一键必须包含 `client_id`

### 华为应用授权

推荐映射：

- `provider = huawei_app`
- `provider_app_id = 华为应用标识`
- `provider_subject = 华为返回的用户唯一主体标识`
- `provider_union_id = NULL`

注意事项：

- 华为厂商返回字段命名可能与微信不同，但统一映射到 `provider_subject`
- 厂商特有资料进入 `profile`，不要继续新增专属核心列

## 典型流程

### 第三方首次授权登录

1. 按 `(provider, client_id, provider_subject)` 查询 `iam.user_identities`
2. 未命中时创建 `iam.users`
3. 新增 `iam.user_identities`
4. 写入成功的 `iam.login_logs`
5. 创建 `iam.user_sessions`

### 第三方重复登录

1. 命中已有 `iam.user_identities`
2. 更新 `last_login_at`
3. 更新 `iam.users.last_login_at`
4. 写入成功日志
5. 轮换或刷新当前 session

### 本地手机号密码登录

1. 按 `(credential_type, principal)` 查询 `iam.user_credentials`
2. 校验密码摘要
3. 更新 `iam.users.last_login_at`
4. 写入成功或失败日志
5. 成功时创建 `iam.user_sessions`

### 绑定手机号

1. 根据当前认证用户定位 `iam.users`
2. 新增或更新 `iam.user_credentials`
3. 同步更新 `iam.users.mobile` 与 `mobile_verified_at`

### 退出登录

1. 按 refresh token 摘要命中 `iam.user_sessions`
2. 设置 `revoked_at`
3. 记录 `revoke_reason`

## 状态与安全约束

### 用户状态

`iam.users.status` 建议复用全局状态码，首版最少支持：

- `ACTIVE(200)`：可登录、可正常使用
- `DISABLED(720)`：禁止登录

### 安全约束

- 不保存明文密码
- 不保存明文 refresh token
- 不保存明文第三方 access token
- 失败日志中的 `reason_detail` 只保存可控文案
- `principal_masked` 只写入脱敏后的主体
- `principal_hash` 应基于归一化后的登录主体计算，不写入手机号、openid、unionid 等明文

## 测试与验收要点

结构层至少覆盖以下场景：

- `iam` schema 可正常创建
- 相同 `(provider, client_id, provider_subject)` 重复插入失败
- 相同 `(credential_type, principal)` 重复插入失败
- 相同 `session_token` 摘要重复插入失败
- 无数据库外键时，非法关联输入仍会被业务层正确拒绝或兜底处理

业务层至少覆盖以下场景：

- 微信首次授权登录成功创建用户、绑定、日志、会话
- 微信重复登录不重复建用户
- 华为首次授权登录成功创建用户、绑定、日志、会话
- 同一用户可同时绑定微信和华为
- 手机号密码登录成功创建会话
- 登录失败仅写日志，不创建会话
- 用户禁用后，密码登录和第三方登录都被拒绝
- refresh token 轮换后，旧会话失效

## 后续落地建议

推荐的实现顺序：

1. 在 `mahu-db` 中新增 `iam` schema 与五张表的 Liquibase 迁移
2. 在 `mahu-entity` 中新增对应实体
3. 在 `mahu-model` 中补充所需查询对象
4. 在 `mahu-dal` 中新增 Repository
5. 在业务模块中补齐第三方登录、会话与日志服务
6. 补充数据库集成测试与服务层测试

若后续决定彻底将登录渠道配置也迁出 `sys`，再单独设计 `iam.auth_clients` 迁移方案，不在本轮设计中一起推进。
