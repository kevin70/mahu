本文是当前阶段的财务域设计稿，用于指导后续 Liquibase、Entity、DAL 与业务服务实现。

## 背景

当前仓库已经补充了独立的 `iam` 认证域设计，用于承载业务用户身份与登录能力。但用户财务信息不适合继续放在 `iam` 或 `sys` 中，原因如下：

1. 财务账户、积分与流水属于独立业务域，后续会继续扩展红包金、赠送金、冻结金等资产类型。
2. 财务数据天然需要更清晰的审计、幂等和并发控制能力，不适合做成用户附属字段。
3. 如果直接在用户表上增加 `balance`、`points` 等列，后续每增加一种资产都需要继续扩列，演进成本高。

因此本方案引入独立的 `wallet` schema，统一承载业务用户财务域。

## 设计目标

本方案需要同时满足以下目标：

- 支持用户余额
- 支持用户积分
- 支持后续扩展更多余额类型
- 支持单账户维度的审计流水
- 支持幂等记账
- 支持冻结与解冻
- 避免为余额、积分分别设计独立主表
- 保持字段命名、主键与状态类型规则清晰稳定

## 总体边界

### `wallet` 与 `iam` / `sys` 的职责边界

- `wallet`：财务账户类型、用户账户、账户流水
- `iam`：业务用户主体、登录、身份绑定
- `sys`：系统管理配置与管理端数据

当前阶段 `wallet` 只通过 `user_id` 关联 `iam.users.id`，不复制用户昵称、手机号等资料。

### 统一建模原则

- 余额、积分和后续资产统一视为不同的账户类型
- 用户当前值与冻结值放在账户表
- 所有变更都必须落流水
- 账户和流水统一使用一套数值字段，不为余额、积分拆出两套列
- 资产语义由账户类型定义，而不是由字段名硬编码
- 低基数、稳定枚举语义的字段优先使用 `SMALLINT`
- 业务编码、业务引用类字段优先使用明确长度的 `VARCHAR(n)`，不默认使用 `TEXT`
- 流水主键使用 `UUID`
- 涉及 `status` 的表字段统一使用 `SMALLINT`
- 首版不引入 `metadata`

## Schema 总览

| 表名 | 用途 |
|---|---|
| `wallet.account_types` | 账户类型定义表 |
| `wallet.accounts` | 用户账户表 |
| `wallet.account_entries` | 单账户流水表 |

## 表设计

### `wallet.account_types`

账户类型定义表，用于定义“这是一种什么资产账户”。

推荐字段：

| 字段 | 类型 | 说明 |
|---|---|---|
| `id` | `INT` | 主键 |
| `code` | `VARCHAR(64)` | 类型编码 |
| `name` | `VARCHAR(128)` | 类型名称 |
| `category` | `SMALLINT` | 业务分类 |
| `value_kind` | `SMALLINT` | 数值模型 |
| `asset_unit` | `VARCHAR(16)` | 计量单位 |
| `precision` | `SMALLINT` | 显示与业务精度 |
| `allow_negative` | `BOOLEAN` | 是否允许负数 |
| `status` | `SMALLINT` | 状态码 |
| `created_at` | `TIMESTAMPTZ` | 创建时间 |
| `updated_at` | `TIMESTAMPTZ` | 更新时间 |

字段约定：

- `category`
  - 首版建议：`10 = balance`、`20 = points`
- `value_kind`
  - 首版建议：`10 = decimal`、`20 = integer`
- `asset_unit`
  - 余额可使用 `CNY`
  - 积分可使用 `point`
- `precision`
  - 余额建议 `2`
  - 积分建议 `0`

首版建议内置至少两类：

- `cash_balance`
  - `category = 10`
  - `value_kind = 10`
  - `asset_unit = CNY`
  - `precision = 2`
- `reward_points`
  - `category = 20`
  - `value_kind = 20`
  - `asset_unit = point`
  - `precision = 0`

约束与索引建议：

- 主键：`account_types_pk`
- 唯一索引：`account_types_code_uidx`
- `status` 默认 `200`
- `allow_negative` 默认 `false`

说明：

- 后续新增 `gift_balance`、`bonus_points`、`frozen_bonus` 等资产时，优先新增账户类型，而不是改账户表结构。

### `wallet.accounts`

用户账户表，表示用户在某一账户类型下的当前账户状态。

推荐字段：

| 字段 | 类型 | 说明 |
|---|---|---|
| `id` | `BIGINT` | 主键 |
| `user_id` | `BIGINT` | 关联用户 |
| `account_type_id` | `INT` | 关联账户类型 |
| `available_value` | `NUMERIC(20,4)` | 当前可用值 |
| `frozen_value` | `NUMERIC(20,4)` | 当前冻结值 |
| `status` | `SMALLINT` | 状态码 |
| `version` | `BIGINT` | 并发版本号 |
| `last_changed_at` | `TIMESTAMPTZ` | 最近一次变更时间 |
| `created_at` | `TIMESTAMPTZ` | 创建时间 |
| `updated_at` | `TIMESTAMPTZ` | 更新时间 |

建模约定：

- 所有账户统一使用 `available_value` / `frozen_value`
- 余额账户通过 `precision = 2` 解释数值
- 积分账户通过 `precision = 0` 解释数值
- 是否允许负数由 `account_types.allow_negative` 控制
- `version` 用于并发控制

约束与索引建议：

- 主键：`accounts_pk`
- 外键：`user_id -> iam.users.id`
- 外键：`account_type_id -> wallet.account_types.id`
- 唯一索引：`accounts_user_id_account_type_id_uidx`
- 普通索引：`accounts_user_id_idx`
- 普通索引：`accounts_account_type_id_idx`
- `available_value` 默认 `0`
- `frozen_value` 默认 `0`
- `status` 默认 `200`
- `version` 默认 `0`

说明：

- 一个用户在一种账户类型下只能有一个账户实例。
- 首版即保留冻结值字段，避免后续补“冻结余额”“冻结积分”时二次改表。

### `wallet.account_entries`

单账户流水表，每条记录描述一次账户变动及变动后的结果。

推荐字段：

| 字段 | 类型 | 说明 |
|---|---|---|
| `id` | `UUID` | 主键 |
| `account_id` | `BIGINT` | 关联账户 |
| `user_id` | `BIGINT` | 关联用户 |
| `account_type_id` | `INT` | 关联账户类型 |
| `reference_type` | `VARCHAR(64)` | 业务关联类型 |
| `reference_id` | `VARCHAR(128)` | 业务关联 ID |
| `direction` | `VARCHAR(16)` | 变动方向 |
| `status` | `SMALLINT` | 状态码 |
| `change_value` | `NUMERIC(20,4)` | 本次变动值 |
| `before_available_value` | `NUMERIC(20,4)` | 变动前可用值 |
| `after_available_value` | `NUMERIC(20,4)` | 变动后可用值 |
| `before_frozen_value` | `NUMERIC(20,4)` | 变动前冻结值 |
| `after_frozen_value` | `NUMERIC(20,4)` | 变动后冻结值 |
| `remark` | `VARCHAR(512)` | 备注 |
| `created_at` | `TIMESTAMPTZ` | 创建时间 |

建模约定：

- `direction`
  - 首版固定 `credit` / `debit`
- `change_value`
  - 始终存正数，增减方向由 `direction` 表达
- `status`
  - 首版建议最少支持：`200` 生效、`930` 已取消
- `reference_type + reference_id`
  - 用于业务关联和幂等
  - 例如：`topup`、`order_pay`、`refund`、`points_reward`

约束与索引建议：

- 主键：`account_entries_pk`
- 外键：`account_id -> wallet.accounts.id`
- 外键：`user_id -> iam.users.id`
- 外键：`account_type_id -> wallet.account_types.id`
- 唯一索引：`account_entries_account_id_reference_type_reference_id_uidx`
- 普通索引：`account_entries_account_id_idx`
- 普通索引：`account_entries_user_id_idx`
- 普通索引：`account_entries_account_type_id_idx`
- 普通索引：`account_entries_created_at_idx`
- `status` 默认 `200`

说明：

- 流水表不做软删除，采用追加写入。
- 若后续允许同一业务生成多条步骤流水，再补 `reference_line` 或调整唯一键。

## 数值与状态规则

### 单值字段设计

本方案不再为余额和积分拆两套字段，而是统一使用：

- `available_value`
- `frozen_value`
- `change_value`

原因如下：

1. 表结构更稳定，后续新增资产类型不需要继续加列。
2. 余额和积分的差异由 `value_kind + precision` 表达，更符合“账户类型驱动”的设计目标。
3. 账户与流水的查询、记账、冻结逻辑可以复用一套实现。

### 数值规则

- 余额类账户：
  - `value_kind = decimal`
  - `precision = 2`
- 积分类账户：
  - `value_kind = integer`
  - `precision = 0`
- 所有值字段统一使用 `NUMERIC(20,4)`
- 业务层必须根据 `value_kind` 校验输入值是否合法

### 状态规则

首版 `status` 字段统一使用 `SMALLINT`，但语义仍建议复用全局状态码：

- `200`：生效
- `720`：禁用
- `930`：已取消

其中：

- `account_types.status`：决定该类型是否可继续建户或记账
- `accounts.status`：决定该账户是否可继续操作
- `account_entries.status`：决定该流水是否生效或已取消

## 典型流程

### 自动建户

1. 按 `(user_id, account_type_id)` 查询 `wallet.accounts`
2. 未命中时创建账户
3. 初始化 `available_value = 0`、`frozen_value = 0`

### 余额充值

1. 按账户类型定位余额账户
2. 校验 `reference_type + reference_id` 幂等
3. 记录 `account_entries`
4. 更新 `accounts.available_value`
5. 更新 `last_changed_at` 与 `version`

### 积分发放

1. 定位积分账户
2. 校验输入值符合 `integer + precision = 0`
3. 记录流水
4. 更新账户可用值

### 余额或积分扣减

1. 校验账户状态与账户类型状态
2. 校验是否允许负数
3. 写入扣减流水
4. 更新账户当前值

### 冻结与解冻

1. 冻结时减少 `available_value`，增加 `frozen_value`
2. 解冻时减少 `frozen_value`，增加 `available_value`
3. 两类操作都必须写入流水，并记录前后值

## 测试与验收要点

结构层至少覆盖以下场景：

- `wallet` schema 可正常创建
- 跨 schema 外键可正常建立到 `iam.users`
- 同一 `(user_id, account_type_id)` 重复建户失败
- 同一 `(account_id, reference_type, reference_id)` 重复记账失败
- `account_entries.id` 的 UUID 主键可正常生成与插入

业务层至少覆盖以下场景：

- 用户首次获取余额账户时自动建户
- 用户首次获取积分账户时自动建户
- 同一用户可同时拥有 `cash_balance` 与 `reward_points`
- 余额充值成功后账户值与流水前后值一致
- 积分发放成功后账户值与流水前后值一致
- 扣减时在 `allow_negative = false` 下不能小于 `0`
- 冻结与解冻后账户值正确迁移
- 同一业务幂等键重复提交不会重复记账

## 后续落地建议

推荐实现顺序：

1. 在 `mahu-db` 中新增 `wallet` schema 与三张表的 Liquibase 迁移
2. 在 `mahu-entity` 中新增对应实体
3. 在 `mahu-model` 中补充账户与流水查询对象
4. 在 `mahu-dal` 中新增 Repository
5. 在业务模块中补齐余额、积分、冻结与流水服务
6. 补充数据库集成测试与服务层测试

如果后续需要更严格的财务对账模型，再单独评估是否从单账户流水升级到双分录总账，不在本轮设计中提前引入。
