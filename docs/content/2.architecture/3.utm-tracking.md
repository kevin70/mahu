## UTM 是什么

UTM（Urchin Tracking Module）是一组追加在 URL 查询参数中的标识字段，用于记录流量来源、投放媒介和活动信息，方便后续做访问、注册、下单等事件的归因分析。

典型用途：

- 识别流量来自哪个平台或渠道
- 比较不同活动、创意或入口位的效果
- 支持广告投放、邮件营销、社交传播等多渠道归因

示例：

```text
https://example.com/landing?utm_source=google&utm_medium=cpc&utm_campaign=spring_sale_2026&utm_term=running_shoes&utm_content=ad_variant_a
```

## 标准参数

UTM 标准上通常使用 5 个参数。

| 参数 | 是否常用 | 说明 | 示例 |
|---|---|---|---|
| `utm_source` | 必填推荐 | 流量来源，表示平台、站点或渠道来源 | `google`、`wechat`、`newsletter` |
| `utm_medium` | 必填推荐 | 流量媒介或投放方式 | `cpc`、`email`、`social`、`referral`、`organic` |
| `utm_campaign` | 强烈建议 | 活动或推广项目标识 | `spring_sale_2026` |
| `utm_term` | 可选 | 关键词，常用于搜索广告 | `running_shoes` |
| `utm_content` | 可选 | 同一活动内区分不同创意、按钮或位置 | `banner_top`、`button_a` |

## UTM 与设备信息的边界

UTM 用于追踪**流量来源**，不用于追踪**用户设备**。设备、终端、浏览器和操作系统应通过独立字段记录，不应塞进 `utm_source`、`utm_medium` 或其他 UTM 参数。

简单区分如下：

- `utm_source`：用户从哪里来，例如 `google`、`wechat`、`newsletter`
- `utm_medium`：通过什么媒介来，例如 `cpc`、`email`、`social`
- `device / terminal`：用户用什么设备访问，例如 `mobile`、`desktop`、`ios`、`android`

错误示例：

- `utm_source=iphone`
- `utm_medium=ios`
- `utm_campaign=wechat_ios_user`

这些值会把“归因信息”和“设备信息”混在一起，后续很难做稳定统计。

## 设备追踪建议

### 推荐字段

如果要追踪用户使用的设备，建议单独记录以下字段：

| 字段 | 作用 | 示例 |
|---|---|---|
| `user_agent` | 原始请求头，保留完整设备特征 | `Mozilla/5.0 ...` |
| `terminal_type` | 业务终端类型 | `web`、`h5`、`ios`、`android`、`wechat_mp` |
| `device_type` | 设备形态 | `desktop`、`mobile`、`tablet` |
| `os_name` | 操作系统名称 | `ios`、`android`、`windows`、`macos` |
| `os_version` | 操作系统版本 | `17.4`、`14` |
| `browser_name` | 浏览器名称 | `chrome`、`safari`、`wechat` |
| `browser_version` | 浏览器版本 | `123.0` |

### 最低建议

如果当前只准备保留最小字段集，至少保存：

- `user_agent`
- `terminal_type`

其中：

- `user_agent` 负责保留原始证据
- `terminal_type` 负责提供稳定、可聚合的业务维度

### 当前仓库现状

仓库里已经存在以下相关能力：

- 请求上下文元数据接口：`mahu-base/src/main/java/cool/houge/mahu/util/Metadata.java`
- Web 请求元数据实现：`mahu-web/src/main/java/cool/houge/mahu/web/WebMetadata.java`
- 访问/登录日志已保存 `user_agent`
  - `mahu-db/src/main/resources/migrations/V20260324_011__create_admin_access_logs.sql`
  - `mahu-db/src/main/resources/migrations/V20260414_001__create_admin_login_logs.sql`
- 认证客户端已有 `terminal_type`
  - `mahu-db/src/main/resources/migrations/V20260324_005__create_auth_clients.sql`

因此：

- **来源归因**继续使用 UTM
- **设备识别**优先使用 `user_agent + terminal_type`
- 后续若需要更细分析，再补 `device_type / os_name / browser_name`

## 命名规范

### 基本规则

1. 参数值统一使用小写。
2. 单词分隔统一使用下划线，不使用空格。
3. 不在 UTM 中放置个人敏感信息（PII），例如邮箱、手机号、身份证号。
4. 命名优先短、稳、可聚合，不要混入临时说明文字。

### 推荐写法

- `utm_source`
  - 推荐表示平台、域名、合作方或投放来源
  - 示例：`google`、`baidu`、`wechat`、`weibo`、`partner_x`

- `utm_medium`
  - 推荐表示媒介或流量类型
  - 示例：`organic`、`cpc`、`email`、`social`、`referral`、`affiliate`、`offline`

- `utm_campaign`
  - 推荐表示一条稳定的活动标识，而不是自然语言标题
  - 建议模板：`{product}_{goal}_{yyyymm}` 或 `{project}_{action}_{yyyymm}`
  - 示例：`shoes_launch_202601`、`vip_recall_202604`

- `utm_term`
  - 推荐仅在确实存在关键词语义时使用
  - 示例：`running_shoes`

- `utm_content`
  - 推荐用于区分创意、按钮、落点位置或实验版本
  - 示例：`banner_top`、`btn_register`、`variant_b`

### 不建议的命名

- `Google`、`WeChat`：大小写不统一
- `spring sale`：包含空格
- `campaign_for_april_new_user_growth_test_a`：过长且难聚合
- `user_13800138000`：包含敏感信息

3. 如果来源本身是某个更细的站点或子入口，但报表并不关心细分，建议统一收敛到更高层级来源，如 `wechat_mp -> wechat`、`xiaohongshu_note -> xiaohongshu`。
4. 如果来源不可信或未命中白名单，统一收敛到 `unknown`，同时保留原始值。

### `utm_medium` 推荐值参考

`utm_medium` 没有一份所有系统都强制统一的官方枚举，但在归因分析中应尽量收敛为一组稳定、短小、可聚合的值。下面是一份适合 Mahu 直接采用的推荐集合。

## 采集建议

### 采集时机

- 首次进入落地页时采集
- 注册、登录、下单、线索提交等关键转化节点做快照
- App 或深链场景下，进入应用时也应传递或恢复 UTM 参数

### 采集原则

1. 同时保存结构化字段和原始值。
2. 结构化字段用于索引、过滤、聚合。
3. 原始值用于审计、排错和后续重算。
4. 白名单未命中时不阻塞主业务流程。

### 当前仓库建议

当前仓库已经有
`mahu-entity/src/main/java/cool/houge/mahu/entity/Utm.java`
作为应用层 UTM 数据对象。数据库落表时，建议至少包含：

- `utm_source`
- `utm_medium`
- `utm_campaign`
- `utm_term`
- `utm_content`
- `utm_raw`

其中：

- `utm_source` ~ `utm_content` 用于结构化查询和统计
- `utm_raw` 用于保存原始 query string 或原始 JSON 片段

## 存储建议

推荐同时保存两类信息：

### 结构化字段

```sql
utm_source   varchar(128)
utm_medium   varchar(64)
utm_campaign varchar(256)
utm_term     varchar(128)
utm_content  varchar(128)
```

用途：

- 便于索引和聚合
- 便于按来源、媒介、活动做报表
- 便于写白名单、映射规则和后台筛选

### 原始字段

```sql
utm_raw jsonb
```

或：

```sql
utm_query text
```

用途：

- 保留原始输入
- 便于排查采集问题
- 便于后续重新清洗或重算

### 规范化建议

- 入库前做 `trim + lowercase`
- 必要时做来源映射，例如把 `wechat_official`、`wechat_mp` 映射为统一的 `wechat`
- 允许保留原始值，但分析时优先使用规范化后的结构化字段

## 白名单与数据质量

建议对以下字段维护字典或白名单：

- `utm_source`
- `utm_medium`
- `utm_campaign`（至少对重要活动做规范化）

目标不是“阻止一切未知值”，而是：

- 控制脏数据
- 降低同义词和拼写漂移
- 保证报表口径稳定

白名单未命中时的推荐处理：

1. 允许主业务继续，例如允许注册成功、允许下单成功。
2. 原样保存原始值。
3. 结构化值可归一为 `unknown` 或 `other`。
4. 将未命中值纳入后续审核和映射流程。

## 实践建议

- UTM 适合用于营销链路的手动打标。
- 自动广告标识（例如 `gclid`）可以作为补充，但不替代 UTM 规范。
- 同一活动中若有多个按钮或广告位，应使用 `utm_content` 区分。
- 同一会话内要注意归因去重，避免重复计算多个来源。
- 不要把 UTM 当作完整事件模型；它是“入口归因标签”，不是全量埋点替代品。

## FAQ

::accordion

:::accordion-item{label="注册时提供的 utm_source 不在白名单内，是否应该拒绝注册？" icon="i-lucide-circle-help"}
- 不应该。UTM 数据质量问题不应阻断核心转化流程。
- 应允许注册成功，同时保留原始值，并将结构化值落为 `unknown` / `other` 或待映射状态。
- 后续再通过白名单补充、映射规则和报表修正来提升归因质量。

**结论**

- 不要因为 `utm_source` 未命中白名单而拒绝注册。
- 业务优先，数据可回补；转化一旦丢失无法补回。
:::

:::accordion-item{label="是否必须同时保存结构化字段和原始字段？" icon="i-lucide-circle-help"}
- 推荐同时保存。
- 只保存结构化字段，后续很难排查来源值被清洗后的偏差。
- 只保存原始字段，又不利于查询、索引和聚合分析。

**结论**

- 默认同时保存结构化字段和原始字段。
:::
