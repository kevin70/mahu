## 适用对象

- 需要给 Mahu 模块补测试、补质量门槛说明的开发者。
- 需要在本地或 CI 中快速确认当前改动是否越过基本质量线的维护者。

## 基础能力

项目当前已统一集成以下测试与质量能力：

- `JUnit 5`、`Mockito`、`AssertJ`
- `Testcontainers + Liquibase`
- `Instancio`
- `Jacoco`
- `Spotless + Palantir Java Format`
- `SonarLint`

以上能力已在根 `build.gradle` 中统一配置，可直接作为各模块测试实现的基础。

## 质量门槛

建议在 CI 中优先对以下模块逐步引入覆盖率门槛：

- `mahu-entity`
- `mahu-shared`
- `mahu-admin:mahu-admin-infra`
- `mahu-task`

- 初始阈值可先设置为 `40%–50%`
- 先作为 Code Review 参考，再逐步提升为硬门槛
- 统一通过 Jacoco 报告输出结果

当前仓库还没有把这些覆盖率阈值统一落成硬性构建失败条件，因此这里描述的是**推荐门槛**，不是已经生效的 CI 阻断规则。

## 推荐执行顺序

如果你只想快速判断一次改动是否达标，建议按下面顺序执行：

1. 先跑受影响模块的 `test`
2. 再跑一次根任务 `./gradlew test`
3. 最后根据需要补 `spotlessCheck`、`openApiGenerate` 或 `jacocoRootReport`

### 常用 Gradle 命令

在项目根目录执行：

```bash [Terminal]
./gradlew build
./gradlew test
./gradlew :mahu-web:test
./gradlew spotlessCheck
./gradlew :mahu-admin:mahu-admin-web:openApiGenerate
```

## 测试规范入口

测试编写规则、模块级测试选型和命名约定统一见「[工程开发约定](/architecture/engineering-conventions)」。

本页只保留测试基础设施、质量门槛和检查入口，避免与工程规范双份维护。

## 后续改进方向

- 在 CI 中逐步把关键模块覆盖率从“可见”提升到“有门槛”。
- 为 OpenAPI 生成路径建立更系统的接口回归测试。
- 为数据库迁移和实体一致性增加自动化校验脚本。

## 注意事项

- 不要把“本页推荐阈值”误认为“所有模块都已经配置了统一 fail-fast 规则”。
- 如果改动涉及 OpenAPI、Liquibase 或生成代码链路，仅跑单元测试通常不够，需要补相应的生成或构建验证。
