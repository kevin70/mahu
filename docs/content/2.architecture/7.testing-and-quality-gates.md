## 基础能力

项目当前已统一集成以下测试与质量能力：

- `JUnit 5`、`Mockito`、`AssertJ`
- `Testcontainers + Liquibase`
- `Instancio`
- `Jacoco`
- `Spotless + Palantir Java Format`
- `SonarLint`

以上能力已在根 `build.gradle` 中统一配置，可直接作为各模块测试实现的基础。

## 质量门槛

建议在 CI 中优先对以下模块逐步引入覆盖率门槛：

- `mahu-entity`
- `mahu-shared`
- `mahu-admin:mahu-admin-infra`
- `mahu-task`

- 初始阈值可先设置为 `40%–50%`
- 先作为 Code Review 参考，再逐步提升为硬门槛
- 统一通过 Jacoco 报告输出结果

### 常用 Gradle 命令

在项目根目录执行：

```bash
./gradlew build
./gradlew test
./gradlew :mahu-web:test
./gradlew spotlessCheck
./gradlew :mahu-admin:mahu-admin-web:openApiGenerate
```

## 测试选型

- `mahu-dal`：优先写真实数据库集成测试，验证查询、排序、更新和落库行为。
- 依赖 `helidon-service-registry` 的模块：优先写基于 Service Registry 的单测，必要时配合 Mockito 替身。
- `mahu-web`：优先写纯 JUnit 轻量测试，验证异常映射与响应逻辑。
- `mahu-admin-web`：对关键接口补基于 HTTP 的集成测试，作为 OpenAPI 生成链路的回归安全网。

### Helidon Service Registry 单测公共约定

适用于所有已依赖 `helidon-service-registry` 的模块。

- 统一使用 `@Testing.Test(perMethod = true)`，保证每个用例拥有独立的 Service Registry。
- 依赖替身统一通过 `Services.set(...)` 注入，不手工拼装被测对象。
- 测试类一旦使用了 Helidon Testing / Service Registry 注解，就默认强制采用该模式，不回退到手工装配。
- `Config` 优先使用最小真实配置节点；调度类测试将对应调度项设置为 `enabled=false`。
- 带 `@Transactional` 的单测统一复用 `cool.houge.mahu.task.testing.HelpScopeTransTestBase`，避免事务增强触发默认数据库初始化。
- 时间相关断言统一使用固定 `Instant.parse(...)` 输入。
- 每个测试方法保持自包含，不依赖前置用例的 Registry 状态或共享可变数据。

## 后续改进方向

- 在 CI 中逐步把关键模块覆盖率从“可见”提升到“有门槛”。
- 为 OpenAPI 生成路径建立更系统的接口回归测试。
- 为数据库迁移和实体一致性增加自动化校验脚本。
