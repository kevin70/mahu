## 当前现状

项目当前已经具备以下质量基础设施：

- `JUnit 5`、`Mockito`、`AssertJ` 作为基础测试栈。
- `Testcontainers + Liquibase` 用于数据库相关集成测试。
- `Instancio` 用于测试数据构造。
- `Jacoco` 聚合覆盖率报告。
- `Spotless + Palantir Java Format` 用于格式一致性。
- `SonarLint` 用于静态检查。

这些能力已经在根 `build.gradle` 中统一配置，可直接作为各模块测试与质量控制的基础。

## 推荐实践

### 覆盖率与关键模块

项目已经集成了 Jacoco 报告与 SonarLint，推荐在 CI 中对以下模块逐步引入覆盖率门槛：

- `mahu-entity`
- `mahu-shared`
- `mahu-admin:mahu-admin-infra`
- 任务/调度相关模块（如 `mahu-task`）

建议做法：

1. 在 CI 流水线中生成 Jacoco 报告。
2. 针对上述模块设定一个较为保守的初始阈值（例如 40%–50%），随着代码质量提升逐步上调。
3. 将覆盖率结果纳入 Code Review 参考，而不是一开始就作为“硬性阻断条件”，以免影响开发体验。

### 常用 Gradle 命令

在项目根目录执行：

```bash
./gradlew build
./gradlew test
./gradlew :mahu-web:test
./gradlew spotlessCheck
./gradlew :mahu-admin:mahu-admin-web:openApiGenerate
```

### OpenAPI 生成路径的回归保障

管理端使用 OpenAPI 生成控制器与 VO，建议：

- 为关键接口（权限、配置、任务等）编写基于 Helidon WebServer 的集成测试，仅针对公开 HTTP 接口进行验证。
- 在测试中覆盖典型的成功/失败分支，尤其是认证失败、权限不足、参数校验失败等场景。
- 将这类测试视为生成代码的“安全网”，在升级 OpenAPI generator 版本或调整模板时，能第一时间发现行为变化。

通过以上措施，可以在不显著增加日常工作量的前提下，让 Mahu 的整体架构在演进中保持较高的稳定性和可维护性。

## 按模块选择测试方式

- `mahu-dal`：优先写真实数据库集成测试，验证查询条件、排序和更新落库行为。
- `mahu-shared` / `mahu-task`：优先写通过 Helidon Service Registry 注入依赖的单测，必要时配合 Mockito 替身。
- `mahu-web`：优先写纯 JUnit 轻量测试，验证异常映射与响应逻辑。
- `mahu-admin-web`：对关键接口补基于 HTTP 的集成测试，把生成代码链路当作回归安全网。

## 后续改进方向

- 在 CI 中逐步把关键模块覆盖率从“可见”提升到“有门槛”。
- 为 OpenAPI 生成路径建立更系统的接口回归测试。
- 为数据库迁移和实体一致性增加自动化校验脚本。
