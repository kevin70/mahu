## 模块职责总览

- **`mahu-base`**：项目级基础设施与通用工具。
- **`mahu-config`**：配置读取与配置模型封装。
- **`mahu-entity`**：数据库实体模型（与表一一对应）。
- **`mahu-shared`**：跨应用共享的领域/应用服务。
- **`mahu-web`**：通用 HTTP/Web 平台能力。
- **`mahu-admin:mahu-admin-infra`**：管理端后台领域与应用服务。
- **`mahu-admin:mahu-admin-web`**：管理端 HTTP/API 入口（基于 OpenAPI 生成）。
- **`mahu-db`**：数据库 Schema 与变更脚本。

下面针对本次重点优化的几个模块，给出更细的职责与准入边界说明。

## `mahu-base`

**定位**：项目的“基础工具箱”，提供与具体业务无关的基础能力。

- **允许内容**：
  - 与具体业务无关的工具类（如 ID 生成、时间工具、集合/字符串辅助方法等）。
  - 与日志、MDC、traceId 相关但 **不依赖具体业务语义** 的封装。
  - 极少量、无法再向下抽象的核心基础设施（例如版本信息生成）。
- **不鼓励/禁止内容**：
  - 直接依赖某个上层业务模块（如 admin、task 等）。
  - 带有强业务含义的枚举、DTO、实体或 Service。
  - 某个具体框架的重度封装（如只服务于 HTTP 层的工具），这类内容应优先放在对应子模块中（例如 `mahu-web`）。

如果一个工具类只在 Web 层使用，或者明显依赖 Helidon API，优先考虑放在 `mahu-web` 中，而不是 `mahu-base`。

## `mahu-shared`

**定位**：真正跨应用共享的领域/应用服务与通用业务能力。

- **允许内容**：
  - 多个应用（如 admin、task、未来的 user-facing 应用）都会复用的领域服务或应用服务。
  - 基于 `mahu-entity`、`mahu-config` 的通用业务逻辑（如通用的文件存储、缓存包装、标准化的元数据处理等）。
  - 无特定应用前缀（如 `Admin*`、`Task*`）的通用业务组件。
- **不鼓励/禁止内容**：
  - 仅被某一个上层应用使用的业务逻辑，尤其是带有明显 `Admin` 前缀或只在管理端出现的概念（应放在 `mahu-admin-infra`）。
  - 与具体 HTTP 协议/框架强绑定的代码（Controller、Filter 等），应放到 `mahu-web` 或上层应用模块。

判断是否应该进 `mahu-shared` 的一个简单规则：**如果当前只有一个应用用到，并且短期内看不到其他应用会复用，就先放在该应用对应的 infra 中，而不是共享模块。**

## `mahu-admin:mahu-admin-infra`

**定位**：管理端的“后端领域层 + 应用层”，承接实体、配置、共享服务，向上暴露给 Web/API。

- **允许内容**：
  - 仅对管理端开放的领域服务、应用服务、聚合逻辑。
  - 管理端特有的仓储实现、外部系统集成（比如只给运营/管理员使用的接口后面的实现）。
  - 管理端特定的权限模型、角色定义、业务状态机等。
- **不鼓励/禁止内容**：
  - 与 HTTP 层紧耦合的代码（如直接依赖 Helidon `ServerRequest` / `ServerResponse`），这些应置于 `mahu-admin-web` 或 `mahu-web` 中。
  - 明显可复用到其他应用的通用逻辑（例如通用文件上传封装、通用缓存包装等），应考虑上收至 `mahu-shared`。

当你在实现某个管理端功能时，优先将“纯业务”和“外部依赖集成”放在 `mahu-admin-infra` 中，只在 `mahu-admin-web` 里做 HTTP 相关的编解码和路由注册。

## `mahu-web`

**定位**：提供统一的 HTTP/Web 平台能力，而不是某个具体应用的业务实现。

- **允许内容**：
  - 通用的错误处理与错误响应模型（例如 `RestErrorHandler` 及其返回结构）。
  - 请求上下文建模与处理（如 `WebMetadata`、traceId 注入、通用 header 解析等）。
  - 与 Helidon WebServer 紧密相关、但又可供多个应用复用的基础设施（如通用 Filter、`HttpFeature` 实现、认证/授权扩展点接口）。
- **不鼓励/禁止内容**：
  - 仅适用于管理端的业务规则或 header 约定（例如只在 admin 中使用的自定义 header、错误码），这类逻辑应保留在 `mahu-admin-web` 或 `mahu-admin-infra` 中。
  - 具体业务接口（Controller、`HttpService` 的具体实现），这些属于各个应用自己的 web 模块。

未来如果新增用户侧 API 或其他子系统，应优先通过依赖 `mahu-web` 获得统一的 HTTP 平台能力，而不是复制粘贴 admin 的实现。

## `mahu-admin:mahu-admin-web`

**定位**：基于 OpenAPI 生成的服务器端骨架，承载管理端的 HTTP/API 入口。

- **允许内容**：
  - OpenAPI 生成的 `H*Service` 控制器及 DTO/VO 类（位于 `src/main/gen`），负责 HTTP 层的参数解析与响应组装。
  - 极少量与管理端前端强相关的适配逻辑（如为了兼容历史前端行为的请求/响应包装）。
  - 针对管理端的路由注册、应用启动类（例如 `AdminApplication`）等。
- **不鼓励/禁止内容**：
  - 大量业务逻辑堆积在生成的控制器或手写 Controller 中，应将其下沉到 `mahu-admin-infra` / `mahu-shared` 的服务层。
  - 其他应用也会使用到的通用 HTTP 能力或中间件（应迁移到 `mahu-web`）。

一个实践建议是：**将 `H*Service` 视为「HTTP 适配层」，仅负责调用一个或少量的应用服务接口，并进行 DTO ↔ 领域对象/命令对象的转换。**

## `mahu-db` 与 `mahu-entity` 的配合

- `mahu-db` 只负责：
  - 使用 Liquibase formatted SQL 管理 Schema 变更；
  - 为每个 `changeSet` 提供可靠的 `rollback`；
  - 按 Schema/表拆分 changelog 文件，保持结构清晰。
- `mahu-entity` 只负责：
  - 映射数据库表的 EBean 实体，遵循根 `README.md` 中的包命名规范；
  - 保持字段命名/注释与数据库表尽量一致。

修改数据库 Schema 时，推荐流程是：

1. 在 `mahu-db` 中新增或修改对应表的 Liquibase SQL 文件，并写好 `rollback`。
2. 同步更新 `mahu-entity` 中的实体类字段与注释。
3. 通过 CI 或本地工具检查两者是否一致（可在后续引入自动校验机制）。
