## UTM 简介

UTM 是 Urchin Tracking Module 的缩写（来源于早期的 Urchin Analytics，后来被 Google 收购并演化为 Google
Analytics）。在现代网络/营销中，UTM 指一组放在 URL 上的查询参数，用来标识流量的来源和营销活动，便于流量与转化的归因分析。

常用 UTM 参数（五个标准参数）

- `utm_source`：流量来源（必填），例如 google、newsletter、wechat。用于识别流量来自哪个平台/渠道。
- `utm_medium`：媒介/类型，例如 cpc、email、social、referral、affiliate。
- `utm_campaign`：活动名称，用于区分不同营销活动或推广系列，例如 spring_sale_2026。
- `utm_term`：关键字（可选），常用于付费搜索来记录关键词。
- `utm_content`：内容区分（可选），在同一广告/链接有多个变体时用于区分（例如 button_a vs button_b）。

UTM 示例 URL
```
https://example.com/landing?utm_source=google&utm_medium=cpc&utm_campaign=spring_sale_2026&utm_term=running+shoes&utm_content=ad_variant1
```

## 为什么要用 UTM？

- 精确归因：把访问/注册/下单等事件归因到具体的广告、邮件或渠道。
- 统一分析：与 Google Analytics 或自建分析系统结合，按 campaign/source/medium 汇总效果。
- 支持 A/B 与多渠道比对：通过 utm_content / utm_campaign 区分不同创意或测试组。

## UTM 命名规范

- 字母小写、去首尾空格：utm 参数统一小写并 trim（例如：Google -> google）。
- 用下划线或短横分隔单词：推荐使用下划线（spring_sale -> spring_sale）。
- 不要包含敏感信息（PII）。
- 必要参数说明：
  - utm_source（必填推荐）：来源平台/域名/媒体名，例如 google / baidu / wechat / newsletter。
  - utm_medium（必填推荐）：媒介类型，例如 organic / cpc / email / social / affiliate / referral / offline。
  - utm_campaign（强烈建议）：活动/项目标识，格式建议：product_action_yyyymm（如 shoes_launch_202601）。
  - utm_term（可选）：关键字/词组（付费搜索场景）。
  - utm_content（可选）：同一 campaign 的不同创意/位置区分（btn_a / banner_top）。
- campaign 命名建议模板（便于过滤与分组）：
  - {product}{purpose}{yyyymmdd}_{variant} 例如: shoes_launch_20260101_a
- 白名单（示例，建议团队内部维护）：
  - utm_source 白名单（示例）： google, baidu, bing, facebook, instagram, tiktok, wechat, qq, weibo, linkedin, partner_x
  - utm_medium 白名单： organic, cpc, ppc, display, email, social, referral, affiliate, offline, sms
- 存储：单独列 utm_source/utm_medium/utm_campaign/utm_term/utm_content，并同时保存 utm_raw（原始 querystring 或完整 URL）。

## 实践与注意事项（对数据库与数据采集的建议）

- 同时保存结构化字段与原始串：在用户/订单属性表里既存 `utm_source/utm_medium/utm_campaign/...` 字段（便于索引、聚合），也保存
  utm_raw 或原始 query string（JSON）用于审计与重算。
- 规范化命名：约定小写、用下划线或短横，不要包含空格或特殊字符（便于聚合与去重）。
- 定义白名单：对 `utm_medium`、`utm_source`、`utm_campaign` 等维护字典，避免脏数据（如同一来源被写成多种拼写）。这就是把
  channel_dim
  用作渠道维度的原因之一。
- 长度与编码：URL 参数要进行 URL 编码；数据库字段适当设置长度（例如 VARCHAR(128) 或 JSONB）；注意一些 URL 长度限制（浏览器/代理）。
- 自动打标 vs 手动打标：广告平台（如 Google Ads）可用自动标签（gclid）配合平台 API 做归因；UTM 常用于手动标记营销邮件、外部媒体、社交等。
- 不在 UTM 中放敏感信息或个人数据（PII）：如邮箱、手机等，避免合规问题。
- 移动 App 场景：App 内跳转或深链也要把 UTM 或同类参数带上，或通过安装时的渠道参数（install referrer）记录。
- 防止重复计数：在会话/转化归因时注意去重（例如同一会话内多次点击不应重复算成多个来源）。

## 存储建议

- user_registration_attrib / order_attrib：

  - 单独列： `utm_source VARCHAR(128)`, `utm_medium VARCHAR(64)`, `utm_campaign VARCHAR(256)`, `utm_term VARCHAR(128)`,
    `utm_content VARCHAR(128)`
  - 原始保存： `utm_raw JSONB` 或 `utm_query TEXT`（保存完整 query string）
  - 索引：对常用字段建立索引（如 utm_source、utm_campaign），对 JSONB 建 GIN 索引以便复杂查询。
  - 快照与规范化：保存原始值同时写入规范化值（小写、trim、映射到 channel_dim）。

## FAQ

::accordion

:::accordion-item{label="注册时提供的 utm_source 未在白名单内，应该让用户注册成功吗？" icon="i-lucide-circle-help"}
- 业务优先：注册转化比关键，直接阻止用户注册会造成流失。归因可以事后修正，但流失无法挽回。
- 数据完整性：必须同时保存原始值（utm_source_raw / utm_raw），以便审计与后续重算。
- 可操作性：将未命中值标记并纳入“映射待审”流程——定期（或触发式）把最常见的未知来源加入白名单或建立映射规则，并在必要时批量重算历史归因。
- 安全与合规：对 utm 值做长度/字符校验与脱敏（不得存 PII），防止 SQL 注入或恶意 payload。

**结论**

- 不要因为 utm_source 不在白名单就阻止用户注册——这会直接降低转化率。
- 采集流程里应允许注册成功，同时把原始 utm_source 原样保存、做规范化并把未命中白名单的值标记为“unknown/other”，并把未命中项上报到映射审核/统计系统以便后续补充与回溯。
